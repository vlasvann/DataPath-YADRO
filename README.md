# TapeSort

## Описание

`TapeSort` — C++20-проект, реализующий сортировку данных на ленточном устройстве.

Программа получает входную бинарную ленту, выходную бинарную ленту и JSON-конфиг. Данные с входной ленты сортируются по возрастанию и записываются на выходную ленту.

Проект моделирует ленточное устройство: чтение и запись выполняются в текущей ячейке, а перемещение по ленте выполняется отдельными операциями.

## Формат хранения данных

Для хранения ленты выбран бинарный формат.

Каждый элемент ленты — `std::int32_t`, то есть одна ячейка занимает 4 байта.

```cpp
offset = position * sizeof(std::int32_t);
```

Текстовый формат для самой ленты не используется, потому что разные числа занимают разное количество символов.

Для удобства тестирования в проект добавлены отдельные утилиты:

```text
TextToBinary
BinaryToText
```

## Модель ленты

Класс `FileTape` реализует интерфейс ленточного устройства поверх бинарного файла.

Основные операции:

```text
Read       — чтение текущей ячейки
Write      — запись в текущую ячейку
MoveRight  — сдвиг ленты вправо
MoveLeft   — сдвиг ленты влево
Rewind     — перемотка в начало
```

`Read()` и `Write()` не изменяют логическую позицию ленты. Для перехода к следующей ячейке нужно явно вызвать `MoveRight()`.

Головка чтения/записи неподвижна, а перемещается сама лента.

Внутри `FileTape` хранится логическая позиция текущей ячейки. Для ускорения последовательного чтения и записи класс также кэширует физические позиции файловых указателей чтения и записи. Не выполняется лишний `seek` на каждом шаге.

## Алгоритм сортировки

Для сортировки используется внешняя сортировка слиянием.

Алгоритм состоит из двух этапов.

### 1. Создание начальных отсортированных блоков

Входная лента читается блоками, размер которых ограничен значением `memory_limit_bytes` из конфига.

Каждый блок:

```text
читается с входной ленты;
сортируется в оперативной памяти;
записывается во временный run-файл.
```

`Run` — это временный отсортированный блок данных.

### 2. K-way merge

После создания начальных run-ов выполняется многопутевое слияние.

Используется `k-way merge`: за один проход сливается сразу несколько отсортированных run-ов.

Для выбора минимального текущего элемента используется min-heap.

Это уменьшает количество проходов по временным данным по сравнению с обычным попарным слиянием.

В памяти одновременно находятся:

```text
буфер начальной сортировки;
текущие элементы открытых run-ов;
небольшие служебные структуры.
```

Временные run-файлы хранятся во внешнем файловом хранилище и не считаются частью оперативного лимита `M`.

## JSON-конфиг

Пример `data/config.json`:

```json
{
  "memory_limit_bytes": 1048576,
  "temp_dir": "tmp",
  "delays": {
    "read_ms": 0,
    "write_ms": 0,
    "move_ms": 0,
    "rewind_ms": 0
  }
}
```

Поля:

```text
memory_limit_bytes  # лимит оперативной памяти для буфера сортировки
temp_dir            # директория для временных run-файлов
read_ms             # задержка чтения
write_ms            # задержка записи
move_ms             # задержка перемещения
rewind_ms           # задержка перемотки
```

`temp_dir` — это директория для временных лент. По умолчанию можно использовать локальную папку `tmp`, не обязательно системная `/tmp`.

## Запуск

Подготовить бинарную входную ленту из текстового файла:

```bash
./build/text_to_binary data/input.txt data/input.bin
```

Запустить сортировку:

```bash
./build/tape_sort data/input.bin data/output.bin data/config.json
```

Перевести результат обратно в текстовый файл:

```bash
./build/binary_to_text data/output.bin data/output.txt
```

Посмотреть результат:

```bash
cat data/output.txt
```

## Особенности реализации

Ограничение `memory_limit_bytes` применяется к основному буферу данных, который используется при создании начальных отсортированных блоков.

Служебная память объектов, путей, файловых потоков и небольших структур слияния не учитывается как часть основного буфера сортировки.

Временные run-файлы являются аналогом временных лент и хранятся на диске.
